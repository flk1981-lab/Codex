# 博士论文项目长期记忆

## 1. 项目对象与当前状态

- 项目：博士毕业论文定稿修订，主题围绕 `SASP/xCT` 在耐药肺结核中的临床验证、机制探索与精准辅助治疗。
- 当前工作重点已经从“大修结构”转入“定稿级收尾”：
  - 证据边界统一
  - 参考文献系统重整
  - 答辩口径收束
  - 细节体例一致性清理

## 2. 内容修改的核心原则

- 优先把论文写成“博士答辩定稿口径”，而不是“投稿论文式强调结果”。
- 对临床结果的表述要始终克制：
  - 可以写“积极信号”“可行性依据”“值得进一步验证”
  - 避免写成“已证明优效”“可直接推荐”
- 对机制结果的表述要始终统一：
  - 第五章是“功能支持 + 机制线索”
  - 不能写成“xCT 机制闭环已经完成终证”
- 总讨论必须同时纳入三条线：
  - 临床信号
  - 宿主侧功能支撑
  - 宿主异质性/精准化探索
- 长期毒性与长期功能结局不是边角料，必须作为核心讨论内容之一。

## 3. 文献处理的固定策略

- 主体论文与文献综述必须分成两套独立编号系统处理，不能混用。
- 文献管理主工具可直接使用本机运行中的 `Zotero`，但最终论文中的引用与参考文献表仍以手工编号系统为准，避免破坏现有版式。
- 每个正文位置最多补 `1-3` 篇文献，避免堆砌式引用。
- 文献选择顺序：
  - 先选权威指南/高质量综述/核心原始研究
  - 再选与论文具体论断直接对应的研究
  - 最后才补背景性文献
- 优先补的段落类型：
  - 讨论段中的判断句
  - 总结段中的概括性结论
  - 前言中具有方向性或理论性的重要铺垫
- 已验证的最佳策略是：
  - 先建立候选文献库
  - 再按“主体/综述”分流
  - 再做逐轮小批量插引
  - 每轮之后都检查“正文最大引用号 = 参考文献末号”

## 4. 本项目已形成的参考文献工作流

- 已建立本地文献库：
  - `output/references/reference_library.json`
  - `output/references/reference_library.bib`
  - `output/references/review_collection.bib`
  - `output/references/body_collection.bib`
- 已验证可用的 Zotero 接入方式：
  - 使用本地 `Zotero Connector import` 端口导入 BibTeX
  - 不依赖 Zotero Local API
- 已形成的检索/入库脚本：
  - `tools/build_reference_library.py`
  - `config/reference_queries.json`
- 以后继续补文献时，优先复用现有库和脚本，不要从零重建。

## 5. DOCX 修改的经验

- 这篇论文更适合“程序化小步写回 + 高频核验”，不适合一次性大改。
- 每次改 `.docx` 之前都要先备份到 `output/backups`。
- 处理 Word 文档时，段落索引会随着插入不断漂移，因此：
  - 不要长期依赖固定段落号做最终判断
  - 最终定位应尽量用标题文本或唯一段落文本
- 对第七章这类已有少量乱码/不稳定段落的区域，优先采用：
  - 在稳定标题后新增“干净段落”
  - 少碰原有不可读段
- 每轮修改后必须做三类核验：
  - 目标段落文本是否真的写入
  - 引文编号是否闭合
  - 参考文献区是否连续、无多余空行、无机械格式错误

## 6. 已经验证有效的章节处理偏好

- 第一章：
  - 重在交代疾病负担、HDT 理论基础、SASP 再定位依据、宿主遗传异质性
  - 引文应偏“指南 + 高质量综述 + 关键原始研究”
- 第三、四章：
  - 强调真实世界前瞻性临床信号
  - 始终提醒非随机、非盲法、样本量有限
  - 长期毒性和长期功能结局必须保留较高权重
- 第五章：
  - 重点是“宿主侧功能支持”
  - 不能把它写成终证性分子机制论文
- 第六章：
  - 定位是“探索性遗传学线索”
  - 重点是提出分层假说，不是建立临床分层规则
- 第七章：
  - 是全篇证据边界统一的总阀门
  - 必须把“临床可继续推进”与“机制尚未闭环”同时写清

## 7. 当前已知的遗留问题

- 文档中仍有少数历史阶段留下的乱码段落，尤其集中在部分章节和综述后段。
- 这些乱码目前没有阻断参考文献系统闭环，但如果继续做最终提交版，建议单独进行一次文字修复。
- 以后若再做“最终定稿清理”，优先顺序应是：
  - 乱码修复
  - 章节口径再统一
  - 表图说明与正文一致性
  - 最终目录/编号/版式复核

## 8. 与用户协作的偏好

- 用户倾向于“你直接做，不用我反复操作软件”。
- 最有效的协作方式是：
  - 我先直接完成一轮
  - 再用一句话说明本轮重点和下轮最值得做的部分
- 用户认可分轮推进，因此后续可继续用“第 N 轮”方式推进，但每轮都应有明确主题，避免泛泛而改。

## 9. 以后继续本项目时应默认继承的约定

- 默认继续在这两份终稿上工作：
  - `output/final/博士毕业论文【FL】2026-04-08-终稿草案.docx`
  - `output/final/博士毕业论文【FL】2026-04-08-终稿草案-含后置页.docx`
- 默认维持两套参考文献系统独立。
- 默认优先保护现有编号和版式稳定性。
- 默认所有新增论断都要服从“证据边界清楚、表述不过强”的总原则。
